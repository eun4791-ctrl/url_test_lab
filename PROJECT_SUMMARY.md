# 프로젝트 분석: QA 자동화 대시보드

## 개요
이 프로젝트는 웹 애플리케이션을 자율적으로 테스트하도록 설계된 AI 기반 QA 자동화 도구입니다. 최신 기술 스택을 활용하여 테스트 케이스를 생성하고, Playwright를 통해 실행하며, 시각 및 텍스트 피드백을 제공합니다.

## 주요 장점

### 1/ AI 기반 자동화
- **자율 테스트 생성**: OpenAI (GPT-4o)를 사용하여 DOM 구조를 분석하고 의미 있는 스모크 테스트(Smoke Test)를 자동으로 생성합니다.
- **스마트 검증**: 모든 케이스에 대해 수동으로 스크립트를 작성할 필요 없이 테스트 로직과 검증 기준을 추론할 수 있습니다.

### 2/ 멀티 디바이스 시각적 검증
- **반응형 테스트**: 데스크톱(1920px), 태블릿(768px), 모바일(375px) 뷰포트에서 동시에 스크린샷을 캡처합니다.
- **시각적 피드백**: 다양한 기기에서 사이트가 어떻게 렌더링되는지 즉시 확인할 수 있습니다.

### 3/ 최신 기술 스택
- **프론트엔드**: React 19, TypeScript, TailwindCSS를 사용하여 반응형이고 타입 안정성이 보장된 UI를 제공합니다.
- **백엔드**: Express + tRPC를 사용하여 클라이언트와 서버 간의 타입 안전한 API 통신을 지원합니다.
- **도구**: Vite를 사용하여 빠른 번들링과 개발 환경을 제공합니다.

### 4/ 설정 불필요 로컬 랩 (Zero-Config)
- **간편한 설정**: `pnpm dev` 명령어로 로컬에서 바로 실행할 수 있으며, OpenAI API 키만 있으면 됩니다.
- **데이터베이스 오버헤드 없음**: 현재 복잡한 데이터베이스 설정 없이 로컬 파일을 사용하여 단순하게 작동합니다.

## 현재 한계점

### 1/ 데이터 영속성 및 확장성
- **로컬 파일 시스템 의존**: 리포트, 스크린샷, 비디오가 로컬 디스크(`/reports`, `/screenshots`)에 직접 저장됩니다.
    - **문제**: Vercel이나 AWS Lambda 같은 서버리스 플랫폼에 배포할 경우, 인스턴스 재시작 시 파일이 모두 손실됩니다 (임시 스토리지).
    - **문제**: 과거 데이터가 신뢰성 있게 보존되지 않으며, 새로운 실행이 이전 리포트를 덮어씁니다.

### 2/ 단일 사용자 및 로컬 중심 설계
- **멀티 테넌시 부재**: 이 앱은 단일 사용자 도구로 설계되었습니다. 다른 사용자의 데이터를 분리하기 위한 사용자 인증 시스템이 없습니다.
- **전역 싱글톤 상태 (동시성 문제)**: 서버(`testRunner.ts`)가 단일 `currentState` 변수를 사용합니다. **전 세계에서 한 번에 하나의 테스트만 실행할 수 있습니다.** 누군가 테스트를 실행 중일 때 다른 사용자가 테스트를 시작하면 차단되거나 오류가 발생합니다.

### 3/ 보안 위험
- **공개 아티팩트**: 모든 리포트와 스크린샷은 인증 없이 `express.static`을 통해 제공됩니다. URL을 아는 사람은 누구나 테스트 결과를 볼 수 있습니다.
- **속도 제한 없음 (Rate Limiting)**: 악의적인 사용자가 테스트 트리거를 스팸처럼 보내 OpenAI API 할당량을 고갈시킬 수 있는 보호 장치가 없습니다.

### 4/ 제한된 기록 및 분석
- **데이터베이스 부재**: 결과가 구조화된 데이터베이스(MySQL/Postgres)에 저장되지 않으므로, 과거 추세(예: "지난달 패스율 보기")를 쉽게 조회할 수 없습니다.
- **파일 덮어쓰기**: 현재 로직은 실행할 때마다 `tc-report.json` 및 스크린샷 파일을 덮어쓰는 경향이 있어, 회귀(Regression) 추적이 어렵습니다.

### 5/ AI 환각 (Hallucinations)
- **오탐/미탐**: 최근 디버깅에서 확인된 것처럼, AI가 때때로 존재하지 않는 셀렉터(`video[playing]`)를 지어내거나 없는 로직을 가정하는 경우가 있어 프롬프트 조정이 필요합니다 (최근 개선됨).

## 프로덕션 도입을 위한 제안
"로컬 랩"을 넘어 상용 서비스 수준으로 발전시키기 위해서는 다음 작업이 필요합니다:
1.  **데이터베이스 통합**: 기존 Drizzle 설정을 활용하여 테스트 결과와 사용자 데이터를 영구적으로 저장해야 합니다.
2.  **클라우드 스토리지**: 스크린샷/비디오를 로컬 디스크 대신 S3 등으로 업로드해야 합니다.
3.  **작업 큐 (Job Queue)**: BullMQ 같은 큐를 사용하여 테스트 실행 요청을 관리함으로써 부하 시 서버 충돌을 방지해야 합니다.
